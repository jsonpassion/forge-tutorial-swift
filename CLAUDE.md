## 콘텐츠 작성 규칙 (가장 중요!)

### 톤 & 스타일
- **언어**: 한국어 (기술 용어는 영문 병기. 예: "옵셔널(Optional)", "구조체(struct)")
- **화자**: "~합니다" 존댓말 사용, 친근하지만 전문적인 톤
- **대상 독자**: 프로그래밍 기초는 있지만 Swift와 iOS 개발은 처음인 개발자
- **핵심 원칙**: "초등학생도 비유로 이해할 수 있게, 하지만 시니어 개발자도 깊이에 만족할 수 있게"
- **어투 가이드**: 딱딱한 교과서체가 아닌, **옆에서 설명해주는 선배** 느낌으로 작성
  - "~입니다" 일변도 대신 "~인데요", "~거든요", "~죠" 등 자연스러운 종결어미 적절히 혼용
  - 독자에게 질문을 던지는 구문 활용: "왜 그럴까요?", "어떤 차이가 있을까요?"
  - 놀라운 사실이나 반전을 만나면 감탄/공감 표현: "놀랍게도", "사실은", "의외로"
  - 단, 과도한 구어체나 반말은 금지. 존댓말 기반의 친근한 톤 유지

### 비유와 예시 규칙
- 모든 새로운 개념에는 반드시 **일상적 비유**를 먼저 제시
- 비유 → 실제 개념 → 코드 순서로 설명
- 비유 예시:
  - 옵셔널 → "선물 상자: 열어보기 전까지는 안에 뭐가 있는지 모릅니다. 비어있을 수도 있죠"
  - 구조체 vs 클래스 → "구조체는 복사한 서류, 클래스는 공유 문서 링크"
  - @State → "화이트보드: 내용이 바뀌면 보고 있는 모든 사람이 변경을 알아차립니다"
  - async/await → "카페에서 커피 주문: 주문(await)하고 다른 일을 하다가 완성되면 받아가는 것"
  - 프로토콜 → "자격증: '이 능력이 있다'고 보증하는 약속"
  - NavigationStack → "책 더미: 새 페이지를 위에 쌓고(push), 뒤로 가면 맨 위를 꺼내는(pop) 것"
  - SwiftData @Model → "스마트 노트: 적으면 자동으로 저장되고, 검색도 되는 노트"
  - Liquid Glass → "반투명 유리창: 뒤의 콘텐츠가 은은하게 비치면서 계층감을 주는 효과"

- 각 섹션에 **> ⚠️ 흔한 오해** 또는 **> 💡 알고 계셨나요?** 박스를 최소 1개 포함
- 초보자가 자주 혼동하는 개념, 흔히 하는 실수, 직관과 다른 사실 등

- 예시:
  - `> ⚠️ **흔한 오해**: "struct는 간단한 것, class는 복잡한 것에 쓴다" — Swift에서는 struct가 기본이고, class는 참조 공유가 필요할 때만 씁니다.`
  - `> 💡 **알고 계셨나요?**: SwiftUI의 body는 매 프레임 호출되는 게 아닙니다. State가 변경될 때만 다시 계산됩니다.`
  - `> 🔥 **실무 팁**: 강제 언래핑(!)은 100% 확신이 있을 때만 쓰세요. 대부분은 guard let이 더 안전합니다.`
  - `> ⚠️ **흔한 오해**: "@ObservedObject를 @StateObject 대신 쓰면 뷰가 재생성될 때 데이터가 사라질 수 있습니다."`

### 코드 블록 vs 마크다운 구분 (매우 중요!)

앱에서 코드 블록(` ``` `)은 "code" 라벨이 붙고 코드 전용 스타일로 렌더링됩니다.
**실행 가능한 코드만 코드 블록에 넣고, 다이어그램/시각적 설명은 반드시 일반 마크다운으로 작성합니다.**

- **코드 블록 사용 O**: Swift/SwiftUI 실행 코드, 터미널 명령어(swift build 등), Xcode 단축키 설명
- **코드 블록 사용 X**: 개념 다이어그램, 뷰 계층 구조 설명, 데이터 흐름도, 아키텍처 구조도, ASCII art

다이어그램 대체 방법:
- **테이블**: 비교표 (struct vs class), 프로퍼티 래퍼 비교, API 파라미터 설명
- **인용(`>`)**: 부연 설명, 흐름 요약, 팁/주의사항
- **볼드/이모지**: 뷰 계층 표시, 데이터 흐름 방향 (→, ←, ↕️), 상태 표시
- **리스트**: 순서가 있는 흐름/파이프라인, 뷰 라이프사이클

### 특수 코드 블록 태그

ForgeCore의 CodeBlockView는 기본 구문 하이라이팅 외에 특수 언어 태그를 지원합니다.
코드 실행 결과, 에러 메시지, 터미널 출력을 시각적으로 구분하여 표시할 수 있습니다.

| 태그 | 용도 | 스타일 |
|------|------|--------|
| `output`, `result` | 코드 실행 결과 | 회색 배경, 모노스페이스 |
| `error` | 에러 메시지 | 빨간 틴트 배경 |
| `console`, `terminal` | 터미널 출력 | 다크 배경, 초록 텍스트 |
| `run:{언어}` | 인터랙티브 실행 시뮬레이션 | 실행 버튼 + 출력 애니메이션 |

사용 예시:

````markdown
```swift
let numbers = [1, 2, 3, 4, 5]
let doubled = numbers.map { $0 * 2 }
print(doubled)
```

```output
[2, 4, 6, 8, 10]
```

```error
error: Cannot convert value of type 'String' to expected argument type 'Int'
```

```console
$ swift build
Building for debugging...
Build complete! (0.45s)
```
````

#### 인터랙티브 실행 블록 (`run:`)

`run:{언어}` 태그를 사용하면 사용자가 "실행" 버튼을 눌러 코드 실행을 시뮬레이션할 수 있습니다.
바로 다음에 `output` 블록을 배치하면 예상 출력으로 자동 페어링됩니다.

````markdown
```run:swift
let name = "Jason"
print("안녕하세요, \(name)!")
print("Swift를 배워봅시다")
```

```output
안녕하세요, Jason!
Swift를 배워봅시다
```
````

**사용 규칙:**
- `run:` 직후에 `output` 블록을 반드시 배치 (출력이 있는 코드의 경우)
- `print()` 등 텍스트 출력 코드만 사용 (SwiftUI 뷰, GUI 코드 제외)
- 한 섹션에 1~3개 적정 (너무 많으면 학습 흐름 방해)
- 출력이 코드 실행 결과와 정확히 일치해야 함

- **코드 실행 결과**는 주석(`//`) 대신 `output` 또는 `result` 블록으로 분리하여 가독성 향상
- **에러 예시**는 `error` 블록으로 시각적으로 경고 표시
- **터미널/빌드 명령의 출력**은 `console` 또는 `terminal` 블록 사용
- 간단한 한 줄 결과는 기존처럼 주석(`// [2, 4, 6, 8, 10]`)으로 표시해도 무방

### 코드 예제 규칙
- **프레임워크**: Swift + SwiftUI 기준 (UIKit은 브릿지 챕터에서만)
- **Swift 버전**: Swift 6+ 문법 사용 (any 키워드, strict concurrency 등)
- **iOS 버전**: iOS 26+ API 기준 (Liquid Glass, 최신 SwiftUI API)
- **실행 가능**: Xcode에서 복사-붙여넣기로 바로 실행할 수 있어야 함
- **주석**: 핵심 라인마다 한국어 주석 필수
- **출력 포함**: 코드 실행 결과를 주석이나 별도 블록으로 표시
- **점진적**: 간단한 예제 → 심화 예제 순서
- **Preview 포함**: SwiftUI 뷰 코드에는 가능하면 #Preview 매크로 포함

```swift
// 좋은 예시
import SwiftUI

struct ContentView: View {
    // @State로 뷰의 상태를 관리합니다
    @State private var count = 0

    var body: some View {
        VStack(spacing: 20) {
            // 현재 카운트를 표시하는 텍스트
            Text("탭 횟수: \(count)")
                .font(.largeTitle)
                .fontWeight(.bold)

            // 탭하면 카운트가 증가하는 버튼
            Button("탭하세요!") {
                count += 1
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()
    }
}

// Xcode Preview에서 바로 확인 가능
#Preview {
    ContentView()
}
```

## iOS 26 / Liquid Glass 관련 규칙

이 튜토리얼은 iOS 26+을 기본 타겟으로 합니다:

- **Liquid Glass**: Ch4의 마지막 섹션에서 집중적으로 다루되, 이후 챕터에서도 적절히 적용
- **최신 API 우선**: deprecated API 대신 최신 대체 API를 사용
  - `NavigationView` → `NavigationStack`
  - `@ObservedObject` → `@Observable` (가능한 경우)
  - `Core Data` → `SwiftData` (기본)
  - `UIKit` → `SwiftUI` (기본)
- **Liquid Glass 스타일링**: `.glassEffect()`, 머티리얼 관련 수정자를 적극 활용
- **새로운 컴포넌트**: iOS 26에서 추가된 SwiftUI 컴포넌트가 있다면 우선 소개

## 이미지 규칙

### 저장 및 명명
- 경로: `assets/images/` 디렉토리에 저장
- 형식: PNG 권장 (스크린샷, 다이어그램), JPG (사진)
- 명명: `{챕터번호}-{섹션번호}-{설명}.png` (예: `03-01-swiftui-preview.png`)
- **다이어그램은 코드 블록이 아닌 마크다운(테이블, 인용, 리스트)으로 표현**
- Xcode 스크린샷은 다크 모드 기준으로 캡처

### 경로 규칙
ForgeCore가 이미지 경로를 자동 처리합니다:
- **상대경로 사용**: `![alt](images/ch03/preview.png)` → baseURL에 자동 결합
- **파일명만 사용**: `![alt](preview.png)` → `assets/images/preview.png`으로 자동 변환 (레거시 호환)
- **권장 형식**: `![SwiftUI Preview 화면](assets/images/03-01-swiftui-preview.png)`

### 출처 표기
외부 이미지(Apple 공식 문서, WWDC 슬라이드, 블로그 등)를 사용할 때 출처를 표기합니다.
ForgeCore가 마크다운 이미지 title 필드에서 출처를 자동 추출하여 표시합니다.

| 형식 | 마크다운 | 앱 표시 |
|------|----------|---------|
| 출처 없음 | `![설명](path)` | 출처 표시 안 함 |
| URL만 | `![설명](path "https://source-url.com")` | 도메인 자동 추출 (예: `developer.apple.com`) |
| 커스텀 라벨 | `![설명](path "Apple Developer \| https://source-url.com")` | 커스텀 라벨 표시 (예: `Apple Developer`) |

사용 예시:
```markdown
<!-- 출처 없음 (직접 제작한 다이어그램) -->
![뷰 라이프사이클](assets/images/05-01-view-lifecycle.png)

<!-- URL만 — 도메인 자동 추출 표시 -->
![SwiftUI 앱 구조](assets/images/03-01-app-structure.png "https://developer.apple.com/swiftui/")

<!-- 커스텀 라벨 — WWDC 세션 출처 명시 -->
![Liquid Glass 디자인](assets/images/04-05-liquid-glass.png "WWDC25 | https://developer.apple.com/wwdc25/")
```

## manifest.json 형식
```json
{
  "chapters": [
    {
      "id": "chapter-id",
      "title": "챕터 제목",
      "description": "설명",
      "sections": [
        {
          "id": "section-id",
          "title": "섹션 제목",
          "file": "폴더/파일명.md",
          "description": "설명"
        }
      ]
    }
  ]
}
```
manifest.json 구조는 절대 임의로 변경하지 않습니다. 변경이 필요하면 사용자에게 먼저 확인합니다.

### manifest.json 버전 규칙 (Semantic Versioning)
manifest.json의 `version` 필드는 다음 규칙에 따라 관리합니다:

| 버전 변경 | 조건 | 예시 |
|-----------|------|------|
| `1.0.X` → `1.0.(X+1)` | 콘텐츠 수정/추가 시 (섹션 작성, 오타 수정 등) | 1.0.7 → 1.0.8 |
| `1.X.0` → `1.(X+1).0` | 챕터 추가/구조 변경 시 (manifest 구조 변경) | 1.1.0 → 1.2.0 |
| `X.0.0` → `(X+1).0.0` | 대규모 리뉴얼 시 (전체 재구성, 기술 스택 변경 등) | 1.0.0 → 2.0.0 |

- 상위 버전이 올라가면 하위 버전은 0으로 리셋 (예: 1.2.3 → 1.3.0)
- **콘텐츠 커밋 시 반드시 patch 버전(X) 증가**
- `lastUpdated` 날짜도 함께 갱신

## 중요 주의사항
1. **순서 준수**: 반드시 manifest.json의 챕터/섹션 순서대로 작성
2. **일관성**: 모든 섹션이 동일한 구조와 톤을 유지
3. **웹 검색 필수**: 콘텐츠 작성 전 반드시 해당 주제의 최신 자료를 웹 검색
4. **status 업데이트 필수**: 섹션 완료 시 contents_status.md 반드시 갱신
5. **참고 자료 필수**: 모든 섹션에 유효한 외부 링크 포함 (Apple 공식 문서 우선)
6. **코드 블록 구분**: 실행 코드만 코드 블록, 다이어그램/구조 설명은 마크다운 테이블/인용 사용
7. **스토리 필수**: 모든 섹션에 역사적 에피소드, 탄생 배경, Swift/Apple 관련 이야기 등 최소 1개 포함
8. **오해/팁 필수**: 모든 섹션에 흔한 오해, 혼동되는 개념, 실무 팁 중 최소 1개 포함
9. **친근한 어투**: 교과서체를 지양하고, 옆에서 설명해주는 선배 느낌의 자연스러운 톤 유지
10. **iOS 26 기준**: Liquid Glass 디자인 언어와 최신 SwiftUI API를 기본으로 사용
11. **SwiftUI 우선**: UIKit은 브릿지가 필요한 경우에만 사용하고, SwiftUI 방식을 기본으로 안내
12. **개인정보 처리방침 준수**: 콘텐츠에서 외부 API, 분석 도구, 사용자 데이터 수집을 안내할 때 반드시 PRIVACY.md 정책과 일치하는지 확인. 이 앱은 **개인정보를 수집하지 않으며**, 외부 분석 SDK/광고를 사용하지 않고, 로컬 저장만 사용함
13. **앱 이름 통일**: 앱을 지칭할 때는 **Forge: Swift**, 튜토리얼 제목은 **2026: Swift 완전 정복**으로 표기
